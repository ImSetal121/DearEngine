# 栈变量与堆变量

## 先有一个直觉

程序运行时，内存分两个主要区域来存放数据：

| | 栈（Stack） | 堆（Heap） |
|---|---|---|
| 管理者 | 编译器自动管 | 程序员手动管（或智能指针代劳） |
| 速度 | 极快 | 相对慢 |
| 大小 | 有限（通常几 MB） | 很大（受物理内存限制） |
| 生命周期 | 离开作用域自动销毁 | 手动 delete 才销毁 |

---

## 栈变量

```cpp
void foo()
{
    int x = 42;              // 栈变量
    BoxShapeSettings s(...); // 栈变量
} // ← 函数结束，x 和 s 自动销毁，无需任何操作
```

**规律：直接写类型名声明的局部变量，都是栈变量。**

栈的工作方式像一摞盘子：进入函数时往上叠，离开函数时从顶部拿走。这个过程由编译器生成的代码自动完成，速度极快，你不需要操心。

### 优点
- 不会忘记释放，不会内存泄漏
- 分配/释放只是移动一个指针，几乎零开销

### 限制
- 生命周期和作用域绑定，出了花括号就没了
- 大小必须在编译期确定（不能运行时才决定要多大）
- 栈空间有限，不适合存大数组或大对象

---

## 堆变量

```cpp
void foo()
{
    int* p = new int(42);       // 堆变量，p 本身是栈上的指针
    BoxShapeSettings* s = new BoxShapeSettings(...); // 堆变量

    delete p;  // 必须手动释放，否则内存泄漏
    delete s;
}
```

**规律：用 `new` 创建的对象，存在堆上。**

堆像一个大仓库：你可以随时申请任意大小的空间，用完后归还。但仓库不会自动帮你清理，忘记归还就是内存泄漏。

### 优点
- 生命周期灵活，可以跨函数存活
- 可以在运行时决定大小
- 空间大，适合存大型数据

### 限制
- 必须手动释放（或用智能指针）
- 忘记 `delete` → 内存泄漏
- `delete` 两次 → 程序崩溃（double-free）

---

## 用智能指针管理堆

现代 C++ 几乎不直接写裸的 `new`/`delete`，而是用智能指针让堆对象"自动管理"：

```cpp
// unique_ptr：独占所有权，离开作用域自动 delete
std::unique_ptr<BoxShapeSettings> s = std::make_unique<BoxShapeSettings>(...);

// shared_ptr：引用计数，最后一个持有者析构时自动 delete
std::shared_ptr<BoxShapeSettings> s = std::make_shared<BoxShapeSettings>(...);
```

智能指针本身是**栈变量**，但它指向的对象在**堆上**。当智能指针离开作用域时，它的析构函数负责 `delete` 堆上的对象。

---

## Jolt 里的引用计数（Ref<>）

Jolt 自己实现了一套类似 `shared_ptr` 的引用计数，用 `Ref<T>` 和 `RefConst<T>` 表示。

```cpp
// 堆上创建，Ref<> 持有，引用计数 = 1
Ref<BoxShapeSettings> s = new BoxShapeSettings(...);
// s 析构 → 引用计数归零 → 自动 delete ✅
```

**问题出在栈对象上：**

```cpp
BoxShapeSettings s(...); // 栈上

// Jolt 内部某处临时用 Ref<> 持有了 s
// → 引用计数归零时，Ref<> 尝试 delete s   ← 第一次释放（错误！）
// → 函数结束，编译器再次析构 s             ← 第二次释放（崩溃！）
```

解决方案就是 `SetEmbedded()`：

```cpp
BoxShapeSettings s(...);
s.SetEmbedded(); // 告诉引用计数："我在栈上，你别管我的释放"

// 之后引用计数即使归零，也不会调用 delete ✅
// 函数结束，编译器正常析构一次            ✅
```

---

## 快速判断口诀

```
有 new  → 堆，需要管理生命周期
没有 new → 栈，自动销毁，省心
```

实际工程中，堆内存几乎都交给智能指针（`unique_ptr` / `shared_ptr` / Jolt 的 `Ref<>`）管理，裸 `new`/`delete` 配对的写法越来越少见。
