# 反射与序列化系统设计指南（编译期版本）

## 一、整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    使用层                                │
│  SceneSerializer    ComponentEditor    EntityCopier     │
├─────────────────────────────────────────────────────────┤
│                    序列化层                              │
│              JsonSerializer (JSON ↔ TypeTree)           │
├─────────────────────────────────────────────────────────┤
│                    反射层（编译期数据）                   │
│    TypeInfo (constexpr)  FieldInfo (constexpr)  无运行时注册 │
├─────────────────────────────────────────────────────────┤
│                    元数据来源（三选一）                   │
│  A. C++26 反射 API   B. constexpr 手写描述   C. 构建时代码生成 │
└─────────────────────────────────────────────────────────┘
```

**与运行时版本的核心区别**：类型信息在编译期或构建期就已确定，不依赖“首次使用时注册”，无运行时 `TypeRegistry` 的插入逻辑，元数据可放在只读段、无堆分配。

---

## 二、编译期反射的含义与可选路线

### 2.1 什么是“编译期反射”

- **编译期**：类型名、字段名、偏移、大小等在**编译阶段**就固定，或由**构建时工具**生成。
- **无运行时注册**：不需要 `TypeRegistry::Register()`、静态初始化顺序依赖、或首次调用时的 lambda 建表。
- **数据形态**：`TypeInfo` / `FieldInfo` 可以是 `constexpr` 常量、或由代码生成器产出，运行时只读、零分配。

### 2.2 三种实现路线（选一条即可）

| 路线 | 描述 | 适用场景 | 当前 C++ 标准 |
|------|------|----------|----------------|
| **A. C++26 反射** | 用语言/标准库提供的 `std::reflection` 等 API 在编译期遍历类型与成员 | 未来标准落地后 | 提案中，尚未正式发布 |
| **B. constexpr 手写** | 为每个类型手写/宏生成 `constexpr TypeInfo`，用 `offsetof`/`sizeof` 等编译期常量 | 类型不多、希望零代码生成 | C++20 起可行 |
| **C. 构建时代码生成** | 用外部工具（脚本/Clang 插件等）扫描头文件，在构建阶段生成 `.refl.h/.refl.cpp` | 类型多、需要与“手写反射”一致的使用方式 | 任意 C++ 版本 |

下文按 **B（constexpr 手写）** 为主线给出完整设计；A、C 在第四节做对照说明。

---

## 三、学习路径（建议顺序，以 constexpr 版为主线）

### 第1步：理解“编译期”带来的差异

**目标**：同样做到“按名访问字段、遍历字段”，但元数据在编译期就存在。

**你需要实现的能力**（使用方式可与运行时版一致，底层不同）：

```cpp
// 1. 编译期/零成本获取类型描述（无哈希表查找）
constexpr const TypeInfo* info = TypeRegistry::Get<MyComponent>();

// 2. 遍历类型的所有字段（info->fields 是 constexpr 数组或 std::array）
for (const auto& field : info->fields) {
    std::cout << field.name << ": " << field.type_name << "\n";
}

// 3. 通过名称访问/修改字段值（与运行时版相同）
void* ptr = field.GetPtr(&component);
field.SetPtr(&component, &new_value);
```

**思考题**：
1. 若 `TypeInfo` 是 `constexpr`，其成员（如字段名、偏移）能否用 `std::string`？若不能，应用什么类型？
2. “按名称查找类型”在无运行时注册表时如何实现？（提示：模板特化 + 显式列表或代码生成）

---

### 第2步：设计可 constexpr 的核心数据结构

**任务**：设计能在编译期构造的 `FieldInfo` 和 `TypeInfo`（C++20）。

**约束**：`constexpr` 中不能使用动态内存、非 `constexpr` 的 `std::string`/`std::vector`，故用固定容量或 `std::array` + 字符串字面量。

**建议文件**：`src/engine/reflection/TypeInfo.h`

**参考代码**：

```cpp
#include <array>
#include <cstddef>
#include <string_view>

struct FieldInfo {
    std::string_view name;      // 编译期字符串
    std::string_view type_name;
    size_t offset = 0;
    size_t size   = 0;

    constexpr void* GetPtr(void* obj) const {
        return static_cast<char*>(obj) + offset;
    }
    constexpr void SetPtr(void* obj, const void* value) const {
        if (value && size > 0)
            __builtin_memcpy(static_cast<char*>(obj) + offset, value, size);
    }
};

template<size_t N>
struct TypeInfoBase {
    std::string_view name;
    std::array<FieldInfo, N> fields;
    size_t field_count = N;

    constexpr const FieldInfo* begin() const { return fields.data(); }
    constexpr const FieldInfo* end() const   { return fields.data() + field_count; }
};

// 类型别名：按字段数量特化，例如 3 个字段
using TypeInfo = TypeInfoBase<3>;
```

**思考题**：
1. 若不同类型字段数量不同，如何统一“TypeInfo”类型？（提示：用 `std::variant` 或类型擦除的接口 + 固定最大 N）
2. 在非 constexpr 的 SetPtr 里用 `std::memcpy` 即可；在 constexpr 中需用编译器内建或 C++20 的 `std::bit_cast` 等，你的环境是否支持？

### 第3步：为单个类型手写 constexpr TypeInfo

**目标**：不写宏、不写代码生成，先为一个类型手写一份“编译期描述”。

**参考代码**：

```cpp
struct TransformComponent {
    float position_x = 0;
    float position_y = 0;
    float rotation  = 0;
};

// 手写：编译期 TypeInfo，无堆、无静态初始化顺序问题
constexpr std::array<FieldInfo, 3> TransformComponent_fields = {{
    {"position_x", "float", offsetof(TransformComponent, position_x), sizeof(float)},
    {"position_y", "float", offsetof(TransformComponent, position_y), sizeof(float)},
    {"rotation",   "float", offsetof(TransformComponent, rotation),    sizeof(float)},
}};

constexpr TypeInfoBase<3> TransformComponent_typeinfo = {
    "TransformComponent",
    TransformComponent_fields,
    3
};
```

**任务**：为 `TransformComponent` 实现 `GetPtr`/`SetPtr` 的 constexpr 友好版本（若你的 SetPtr 需在运行时调用，可保留 `std::memcpy` 的非 constexpr 实现，仅保证 TypeInfo 本身为 constexpr）。

### 第4步：用宏生成 constexpr TypeInfo（减少重复）

**目标**：保持“编译期常量”性质，用宏生成与手写等价的代码。

**期望用法**：

```cpp
class TransformComponent : public IComponent {
public:
    float position_x = 0;
    float position_y = 0;
    float rotation  = 0;

    REFLECT_CLASS_CONSTEXPR(TransformComponent)
    REFLECT_FIELD(position_x)
    REFLECT_FIELD(position_y)
    REFLECT_FIELD(rotation)
    REFLECT_END_CONSTEXPR()
};
```

**宏展开后的效果**（与手写一致，只是由宏生成）：

```cpp
constexpr static std::array<FieldInfo, 3> GetStaticTypeInfoFields() {
    return {{
        {"position_x", "float", offsetof(TransformComponent, position_x), sizeof(float)},
        {"position_y", "float", offsetof(TransformComponent, position_y), sizeof(float)},
        {"rotation",   "float", offsetof(TransformComponent, rotation),    sizeof(float)},
    }};
}
constexpr static const TypeInfoBase<3>* GetStaticTypeInfo() {
    static constexpr std::array<FieldInfo, 3> fs = GetStaticTypeInfoFields();
    static constexpr TypeInfoBase<3> ti{"TransformComponent", fs, 3};
    return &ti;
}
```

**注意**：字段数量会随 `REFLECT_FIELD` 个数变化，可用 `__VA_ARGS__` 或预处理器计数宏生成 `TypeInfoBase<N>` 的 N；若实现复杂，可先固定 N（如 8），多余槽位填空 FieldInfo。

**参考代码**（简化：固定 3 个字段）：

```cpp
#define REFLECT_CLASS_CONSTEXPR(ClassName) \
    constexpr static auto GetStaticTypeInfoFields() { \
        return std::array<FieldInfo, 3>{{ \
            REFLECT_FIELD_IMPL(ClassName, position_x), \
            REFLECT_FIELD_IMPL(ClassName, position_y), \
            REFLECT_FIELD_IMPL(ClassName, rotation) \
        }}; \
    } \
    static const TypeInfoBase<3>* GetStaticTypeInfo() { \
        static constexpr auto fs = GetStaticTypeInfoFields(); \
        static constexpr TypeInfoBase<3> ti{#ClassName, fs, 3}; \
        return &ti; \
    }
#define REFLECT_FIELD_IMPL(ClassName, field) \
    FieldInfo{#field, /* type_name 可用 typeid 或手写 "float" */ "float", \
              offsetof(ClassName, field), sizeof(decltype(((ClassName*)0)->field))}
```

实际工程中可用 `BOOST_PP_SEQ_FOR_EACH` 或自写计数宏支持任意数量字段。

### 第5步：“TypeRegistry”的编译期版本（无运行时 map）

**目标**：保留 `TypeRegistry::Get<T>()` 的调用方式，但实现为编译期查表，无 `unordered_map`。

**方案 1：仅按类型取信息（不按名字）**

```cpp
template<typename T>
const TypeInfoBase<N>* TypeRegistry::Get() {
    return T::GetStaticTypeInfo();
}
```

不需要全局注册表，每个类型自带 `GetStaticTypeInfo()`。

**方案 2：按名称查找（用于反序列化时 “string → TypeInfo”）**

若需要 `GetByName("TransformComponent")`，可：

- **手写派发**：`if (name == "TransformComponent") return TransformComponent::GetStaticTypeInfo();` 等。
- **编译期表**：用 `std::array<std::pair<std::string_view, const void*>, N>` 存所有类型，在静态存储区构造；查找时线性遍历（类型数不多时可接受）。
- **代码生成**：由构建时工具生成一个 “name → GetStaticTypeInfo” 的 switch 或 map 初始化列表。

**参考代码**（手写派发，不依赖代码生成）：

```cpp
namespace detail {
    inline const void* GetTypeInfoByName(std::string_view name) {
        if (name == "TransformComponent") return TransformComponent::GetStaticTypeInfo();
        if (name == "CameraComponent")     return CameraComponent::GetStaticTypeInfo();
        return nullptr;
    }
}
```

### 示例：编辑器里按字段迭代（与运行时版一致）

使用方式与《反射与序列化系统设计指南》中一致，只是 `TypeInfo` 来自 constexpr，无注册步骤：

```cpp
TransformComponent* comp = /* 当前选中的组件 */;
const auto* ti = TypeRegistry::Get<TransformComponent>();
for (const auto& f : *ti) {
    auto it = g_field_editors.find(f.type_name);
    if (it != g_field_editors.end())
        it->second(f.name.data(), f.GetPtr(comp), f.size);
}
```

按名称改某一字段：先按 `f.name` 找到 `FieldInfo`，再按 `f.type_name` 查表调用对应 Set/Get，与运行时版相同。

### 第6步：JSON 序列化（基于 constexpr TypeInfo）

**任务**：与运行时版相同的接口，但实现中遍历的是 constexpr 的 `fields`。

```cpp
namespace JsonSerializer {
    json Serialize(void* obj, const TypeInfoBase<N>* info);
    void Deserialize(void* obj, const TypeInfoBase<N>* info, const json& j);
}
```

实现思路：遍历 `info->fields`，根据 `field.type_name`（或配合类型 ID）选择序列化/反序列化分支；若 TypeInfo 为 constexpr，遍历本身可内联、零额外开销。

## 四、进阶：C++26 反射与构建时代码生成（对照）

### 4.1 C++26 反射（概念性）

若未来采用 C++26 的反射提案，元数据将来自编译器，无需手写字段列表：

```cpp
// 伪代码，语法以最终标准为准
#include <reflection>

template<typename T>
constexpr auto get_type_info() {
    using namespace std::reflection;
    std::array<FieldInfo, reflexpr(T).members().size()> arr;
    for (size_t i = 0; i < reflexpr(T).members().size(); ++i) {
        auto m = reflexpr(T).members()[i];
        arr[i] = { name_of(m), name_of(type_of(m)), offset_of(m), size_of(type_of(m)) };
    }
    return TypeInfoBase<...>{ name_of(reflexpr(T)), arr };
}
```

届时“编译期版本”可完全由语言支持，无需宏或代码生成。

### 4.2 构建时代码生成

- 在 CMake/脚本中增加一步：运行工具扫描 `*.refl.h` 或带标记的头文件。
- 工具输出 `*.refl.generated.h`：包含每个类型的 `constexpr TypeInfo` 或 `GetStaticTypeInfo()` 实现。
- 工程代码只写 `REFLECT_CLASS(TransformComponent)` 等声明，具体 `FieldInfo` 列表由生成器填写，这样字段数量、类型名、偏移都可自动维护，且仍保持“编译期常量”的形态。

## 六、与运行时版本的对比小结

| 项目         | 运行时版（原文档）     | 编译期版（本文）           |
|--------------|------------------------|----------------------------|
| 元数据构建   | 首次调用时 lambda 建表 | 编译期常量或构建时代码生成 |
| 存储         | 堆 + `TypeRegistry`   | 静态/只读，无堆             |
| 字符串       | `std::string`         | `std::string_view` / 字面量 |
| 按名查类型   | `TypeRegistry::GetByName` | 手写派发或生成表       |
| 适用标准     | C++11/14               | constexpr 版建议 C++20      |
